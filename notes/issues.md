Формат: `[статус]` — открыта / в работе / закрыта

## Решённые

### [закрыта] chip-50 / chip-5 путаница при совместном присутствии
Механизм: FLANN захватывал ключевые точки chip-5 в матчинге для chip-50 → гомография растягивалась между двумя фишками.
**Решение:** добавлена HSV-верификация (постфильтр) + перерегистрация с per_shot=150. chip-5 и chip-25 теперь стабильны.

### [закрыта] chip-25 / chip-5 взаимные ложные срабатывания
Два bbox на каждой. **Решение:** то же — HSV-верификация различает синий и красный.

## Активные проблемы

### [открыта] Нестабильная детекция чёрных объектов (chip-50)
bbox нормальный когда виден, но появляется/пропадает каждые 1–3 мин.
Причины: низкий V → HSV-верификация менее надёжна; глянец даёт блики → нестабильные ключевые точки.
Наблюдение: регистрация в тени (без бликов) возможно улучшит ситуацию — проверить.

### [открыта] chip-1 (белая): очень плохая детекция, bbox как прямая линия
Причины: 1) HSV-фильтр отклоняет белый (близок к фону, бликам); 2) SIFT находит мало уникальных признаков на однородной белой поверхности.
Варианты: расширить HSV_V_TOLERANCE для светлых объектов, или ослабить HSV_VERIFICATION для конкретного профиля.

### [открыта] chip-10 (зелёная): редкие детекции, bbox как прямая, при максимуме признаков (750)
Причина неясна — цвет хорошо отличим, признаков максимум. Гипотеза: регистрация в тени дала другой HSV-профиль центрального кропа, чем при детекции в нормальном освещении.
Проверить: переделать регистрацию chip-10 в нормальном освещении.

### [открыта] Объекты с ≤300 признаками практически не детектируются
lil-blue (266), lil-white (224) — мало текстуры → FLANN не находит достаточно совпадений.
Порог надёжной детекции эмпирически: ≥500 признаков. Ниже — только иногда и нестабильно.
Текущие варианты: увеличить REGISTRATION_FEATURES_PER_SHOT (упирается в реальную текстуру объекта).

### [открыта] "Восьмёрка" — самопересекающийся bbox (chip-10, chip-50)
Нестабильная гомография переворачивает угол → cv2.polylines рисует самопересечение.
Признак ненадёжного матча. Можно отфильтровать по выпуклости bbox или по соотношению сторон.

### [открыта] Деградация chip-25 при 8 зарегистрированных объектах
В сессии 2 (5 объектов) chip-25 была без изъянов. В сессии 3 (8 объектов) — частые пропадания.
Гипотеза: с ростом числа зарегистрированных объектов растёт вероятность ложных FLANN-совпадений для конкретной фишки.

### [открыта] Джиттер bbox при смещении от точки регистрации
Наблюдается у всех фишек. Гомография плохо обобщается на позиции, не покрытые регистрационными снимками.

### [открыта] Освещение при регистрации ≠ освещение при детекции
chip-10 зарегистрирована в тени (без бликов), детектируется в нормальном освещении → HSV-профиль расходится.
Правило: регистрировать в тех же условиях освещения, что будут при игре.

## Производительность

### [открыта] proc time растёт линейно (~70–80 ms/объект)
574 ms при 8 объектах → ~1.7 fps. Критично если объектов станет больше.
Варианты: снизить FLANN_CHECKS, SIFT_FEATURES, добавить ROI-предфильтрацию.

### [открыта] Фокусированная детекция по ходу игрока
При ходе конкретного игрока передавать в `detect_objects` только его фишку (по `active_chip_id`), а не все зарегистрированные объекты. Все 500 точек кадра сосредотачиваются на одном объекте → 200–400 совпадений вместо 15–30 → стабильная детекция даже для chip-1/lil-white.

Реализация:
1. `GameScanner.active_chip_id: str | None = None` — добавить в `__init__`
2. В `tracking_loop` перед `detect_objects`: если `active_chip_id` задан — фильтровать `registered_objects` до одного элемента
3. Смена активной фишки по нажатию клавиши или через HTTP (когда будет очередь ходов)

Требует: очередь ходов (mapping игрок → chip UUID). Логично реализовывать вместе с HTTP-передачей данных хода.
