## Project Description

<!-- TODO: заполнить -->

Система компьютерного зрения для идентификации и отслеживания игровых фишек на поле настольной игры через сканер (OpenCV/SIFT). Фишки регистрируются по визуальным признакам и распознаются в реальном времени; данные о положении передаются для расчёта игрового хода.

Игра типа "из точки А в точку Б" с правилами вида "пропусти ход", "иди на N клеток назад/вперёд". Правила настраиваемы — система должна оставаться гибкой без закладывания конкретики заранее.

## Алгоритмы и техники

### CLAHE (Contrast Limited Adaptive Histogram Equalization)

Применяется перед детектором признаков. В отличие от глобального выравнивания гистограммы, CLAHE делит изображение на мелкие блоки (`CLAHE_TILE_GRID`) и нормализует контраст внутри каждого — с ограничением (`CLAHE_CLIP_LIMIT`), чтобы не усиливать шум. Результат: локальные детали на тёмных/засвеченных участках становятся чёткими, SIFT/AKAZE находит в 2–3 раза больше устойчивых точек на объектах с неравномерным освещением. Параметры: `CLAHE_CLIP_LIMIT` (выше → сильнее контраст, больше шума), `CLAHE_TILE_GRID` (меньше → локальнее эффект).

### Bounding box и центральный кроп

**Bbox (bounding box)** — прямоугольник вокруг обнаруженного объекта. Вычисляется проекцией 4 углов ROI-снимка через гомографию M. Фишка круглая → углы bbox всегда содержат фон (поле, стол). Джиттер bbox — норма: каждый кадр SIFT даёт чуть другой набор инлайеров → M чуть другая → углы смещаются.

**"Восьмёрка"** — самопересекающийся bbox: рёбра пересекаются, образуя фигуру 8 или галстук-бабочку. Возникает когда нестабильная гомография переворачивает один или два угла на противоположную сторону. `cv2.polylines` рисует углы в исходном порядке (0→1→2→3→0), и линии пересекаются. Признак ненадёжного матча — RANSAC нашёл геометрически некорректное решение.

**Центральный кроп** — берётся `HSV_CROP_RATIO` (по умолчанию 0.5) от ширины/высоты bbox вокруг центра. Центральные 25% площади содержат почти только пиксели фишки независимо от фона. Используется для HSV-верификации.

**HSV-верификация** — постфильтр после SIFT. Медиана HSV центрального кропа сравнивается с профилем, записанным при регистрации. Ложные детекции (гомография "уехала" на чужую фишку) отклоняются по цвету. Толерантности: `HSV_H/S/V_TOLERANCE` в settings. Требует перерегистрации фишек, если были зарегистрированы до появления HSV.

### Детектор признаков (SIFT / AKAZE)

Переключается через `settings.DETECTOR_TYPE` ('sift' или 'akaze'). SIFT даёт вещественные дескрипторы (128D float32) — работает с FLANN KD-tree. AKAZE даёт бинарные дескрипторы (M-LDB, uint8) — требует FLANN LSH; более быстрый и лицензионно открытый, но менее устойчив на текстурно-бедных объектах. При смене детектора необходимо перерегистрировать все фишки.

## Setup

**Python:** venv на Python 3.13, зависимости в `requirements.txt` (numpy, opencv-python)

Перед запуском **обязательно** активировать venv:

```bash
# bash / git bash
source .venv/Scripts/activate

# или cmd / powershell
.venv\Scripts\activate
```

Проверить: `which python` должен показывать путь внутри `.venv/`

Установка зависимостей (если new contributor):

```bash
pip install -r requirements.txt
```

**Запуск** (с логированием в `scanner.log`):

```bash
# bash / git bash — прямой путь к python venv, без source activate
PYTHONIOENCODING=utf-8 .venv/Scripts/python -u run_scanner.py 2>&1 | tee scanner.log
```

## Conventions

<!-- TODO: уточнить и дополнить -->

- Все действия управления реализуются двумя способами: клавиша (`cv2.waitKey`) + кнопка мыши в окне просмотра. Исключение — операции с неоднозначным контекстом (например, выделение ROI — только мышь, текстовый ввод — только клавиатура). Новая операция = клавиша + запись в `_compute_button_rects`.
- Классы с инкапсуляцией, контекстные менеджеры где уместно
- Конфигурация только через `config/settings.py` — не хардкодить константы в модулях
- UUID для идентификации объектов (`uuid.uuid4()`)
- Сериализация через pickle (`.pkl`), с резервными копиями перед изменением
- Комментарии и сообщения об ошибках на русском языке
- Новые модули — по образцу существующих (класс + методы, без глобального состояния)
